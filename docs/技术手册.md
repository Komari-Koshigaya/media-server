## 第一章 项目概述

### 1.1 项目定位

Media Server 是一个轻量级局域网媒体服务器，解决的核心问题是：**在 PC 和手机之间共享文件并直接在线播放/阅读，无需安装任何客户端 APP**。

用户只需在 PC 上启动服务，手机通过浏览器访问局域网 IP 即可浏览文件、播放视频、阅读文档、上传文件。服务端零配置启动，密码自动生成，共享目录通过 Web 管理后台动态添加。

### 1.2 核心功能

| 功能 | 说明 |
|------|------|
| 视频/音频播放 | 支持 HTTP Range 拖拽进度条，不兼容格式自动转码 |
| 文档阅读 | TXT（章节检测+搜索高亮）、EPUB（目录+章节切换）、PDF（逐页渲染） |
| 文件上传 | 手机向 PC 上传，实时进度条，最大 10GB |
| 文件浏览 | 目录导航、文件搜索、缩略图预览、一键下载 |
| 管理后台 | Web 界面管理共享目录，可视化文件夹选择器 |
| 鉴权保护 | Cookie + SHA256 token，登录失败限流 |

### 1.3 技术指标

| 指标 | 数值 |
|------|------|
| 源代码总行数 | ~3,200 行（不含测试） |
| 测试代码行数 | ~800 行 |
| 单元测试数量 | 84 个 |
| 模块数量 | 8 个 |
| HTTP 路由数 | 18 条 |
| 支持文件格式 | 65 种（17 视频 + 9 音频 + 35 文本 + 10 图片 + 2 电子书） |
| 最大上传文件 | 10 GB |
| 上传分块大小 | 1 MB |

### 1.4 支持的文件格式

**视频** (17 种)：`.mp4` `.mkv` `.avi` `.mov` `.wmv` `.flv` `.webm` `.m4v` `.ts` `.rmvb` `.3gp` `.mpg` `.mpeg` `.mts` `.m2ts` `.ogv` `.vob`

**音频** (9 种)：`.mp3` `.flac` `.wav` `.aac` `.ogg` `.m4a` `.wma` `.ape` `.opus`

**文本** (35 种)：`.txt` `.md` `.json` `.xml` `.csv` `.log` `.py` `.js` `.html` `.css` `.srt` `.ass` `.sub` `.yaml` `.yml` `.toml` `.ini` `.sh` `.bat` `.sql` `.go` `.java` `.c` `.cpp` `.h` `.rs` `.rb` `.php` `.ts` `.tsx` `.jsx` `.vue` `.lua` `.r` `.swift` `.kt`

**图片** (10 种)：`.jpg` `.jpeg` `.png` `.gif` `.bmp` `.webp` `.svg` `.heic` `.heif` `.avif`

**电子书** (2 种)：`.epub` `.pdf`

### 2.4 数据流

**视频播放请求的数据流**：

```
浏览器 GET /play/0/movie.mp4
  -> 中间件检查 Cookie token
  -> handle_play()
     -> resolve_path() 解析路径（防穿越）
     -> get_file_type() 判断文件类型
     -> async_probe_video_codec() 探测编码
     -> build_player_html() 生成播放器页面
  -> 返回 HTML

浏览器 GET /raw/0/movie.mp4?transcode=1
  -> _handle_transcode()
     -> 创建临时文件
     -> 启动 ffmpeg 子进程
     -> drain_stderr() 协程持续读取 stderr
     -> ffmpeg 完成后 FileResponse 发送
     -> 10 分钟后清理临时文件
```

### 2.5 技术栈选型与理由

| 技术 | 选型理由 |
|------|---------|
| **aiohttp** | 原生 async/await，适合 I/O 密集的文件服务；比 Flask 异步更自然，比 FastAPI 更轻量 |
| **FileResponse** | 底层使用 sendfile 系统调用（零拷贝），自动处理 HTTP Range，无需手动实现分片 |
| **ffmpeg** | 唯一成熟的全格式转码方案；通过 imageio-ffmpeg 获取预编译二进制，免去用户安装 |
| **PyMuPDF (fitz)** | Python 绑定中性能最好的 PDF 渲染库，支持 2x 高清输出 |
| **ebooklib + BeautifulSoup** | ebooklib 解析 EPUB 结构，BS4 清理 HTML 提取纯文本 |
| **Cookie + SHA256** | 无需数据库，局域网场景足够安全；httponly 防 XSS 窃取 |
| **CSS 变量** | 一套变量控制全局暗色主题，修改一处即全局生效 |

## 第四章 模块详解

### 4.1 __init__.py — 日志系统

**职责**：初始化全局日志系统，提供 Windows 控制台兼容的 `safe_print()`。

**设计要点**：

1. **双输出通道**：控制台（INFO 级别）+ 文件（DEBUG 级别）
2. **日志格式**：`%(asctime)s [%(levelname)s] %(message)s`，时间精确到秒
3. **文件位置**：`media_server.log` 在项目根目录，UTF-8 编码
4. **降级策略**：文件创建失败（权限不足）时自动降级为仅控制台输出

```python
import logging, sys

logger = logging.getLogger('media_server')
logger.setLevel(logging.DEBUG)

# 控制台 — INFO 级别，用户可见
_console_handler = logging.StreamHandler(sys.stdout)
_console_handler.setLevel(logging.INFO)
_console_handler.setFormatter(logging.Formatter(
    '%(asctime)s [%(levelname)s] %(message)s', '%Y-%m-%d %H:%M:%S'))

# 文件 — DEBUG 级别，完整记录
_file_handler = logging.FileHandler('media_server.log', encoding='utf-8')
_file_handler.setLevel(logging.DEBUG)

logger.addHandler(_console_handler)
logger.addHandler(_file_handler)
```

**safe_print 的由来**：Windows 控制台默认使用 GBK 编码，直接 `print()` 中文或 emoji 会抛出 `UnicodeEncodeError`。`safe_print()` 捕获此异常，将无法编码的字符替换为 `?`：

```python
def safe_print(msg: str):
    try:
        print(msg)
    except UnicodeEncodeError:
        print(msg.encode('utf-8').decode('gbk', errors='replace'))
```

### 4.2 config.py — 配置管理

**职责**：读写 `shares.json`，管理共享目录列表和密码。

**关键函数**：

| 函数 | 说明 |
|------|------|
| `load_config()` | 加载配置文件，异常时返回空字典（不崩溃） |
| `save_config(cfg)` | 写入 JSON，`ensure_ascii=False` 保留中文 |
| `load_shares()` | 加载共享列表，**自动过滤路径不存在的条目** |
| `save_shares(shares)` | 保存共享列表，**保留已有密码不被覆盖** |
| `get_password()` | 返回已有密码，或自动生成 4 位数字密码 |

**异常处理策略**：所有 I/O 操作都捕获 `FileNotFoundError`、`json.JSONDecodeError`、`PermissionError`、`OSError`，确保配置文件损坏或缺失时服务仍能启动。

### 4.3 auth.py — 鉴权与限流

**职责**：Token 生成和基于 IP 的登录失败限流。

**make_token(password)**：

```python
def make_token(password: str) -> str:
    """SHA256('media_server:{password}') 的前 32 位十六进制"""
    return hashlib.sha256(f'media_server:{password}'.encode()).hexdigest()[:32]
```

**RateLimiter 设计**：

```
IP 记录结构 (_IPRecord):
  +-- fail_count: int     # 连续失败次数
  +-- lock_until: float   # 锁定截止时间戳
  \-- last_active: float  # 最后活动时间

限流策略:
  - 5 次连续失败 --> 锁定 5 分钟
  - 成功登录 --> 清除所有失败记录
  - 30 分钟无活动 --> 自动清理记录（防止内存泄漏）

清理机制:
  - 每次 check() 调用时触发 _cleanup_expired()
  - 单次最多清理 50 条记录（防止瞬时卡顿）
  - 按 last_active 排序，优先清理最久未活跃的
```

### 4.4 utils.py — 工具函数

**职责**：文件类型检测、MIME 映射、格式化、缩略图生成、EPUB/PDF 解析。

**常量定义**：

| 常量 | 值 | 说明 |
|------|---|------|
| `VIDEO_EXTS` | 11 个扩展名 | 视频文件扩展名集合 |
| `AUDIO_EXTS` | 8 个扩展名 | 音频文件扩展名集合 |

## 第五章 核心功能实现

### 5.1 视频转码策略链

**问题背景**：手机浏览器只支持 H.264 视频 + AAC 音频编码。用户在局域网中播放 MKV、AVI 等格式时，需要服务端实时转码。

**设计原则**：速度优先，牺牲暂停恢复精度换取即时播放体验。

**三级策略链**：

```
探测视频/音频编码
  |
  +-- H.264 + AAC -----> 策略1: -c copy（全部复制，秒级完成）
  |
  +-- H.264 + 非AAC ---> 策略2: -c:v copy -c:a aac（只转音频，快）
  |
  \-- 非H.264 ---------> 策略3a: 先试 -c:v copy -c:a aac
                              |
                              \-- 失败 -> 策略3b: -c:v libx264（全量重编码，慢）
```

**ffmpeg 公共参数**（错误恢复）：

```
-fflags +discardcorrupt+genpts+igndts
-err_detect ignore_err
-avoid_negative_ts make_zero
-max_muxing_queue_size 4096
```

这些参数处理损坏帧、缺失时间戳、队列溢出等边缘情况，提高转码成功率。

**临时文件管理**：转码结果写入 `tempfile.mkstemp()` 创建的临时文件，通过 `asyncio.ensure_future` 延迟 10 分钟清理，确保 FileResponse 完成传输后再删除。

### 5.2 HTTP Range 与零拷贝传输

**为什么用 FileResponse 而不是 StreamResponse**：

| 特性 | FileResponse | StreamResponse |
|------|-------------|----------------|
| 零拷贝 | sendfile 系统调用 | 需要手动读写 |
| HTTP Range | 自动处理 206 | 需要手动实现 |
| 拖拽进度条 | 原生支持 | 需要额外代码 |
| 内存占用 | 几乎为零 | 需要缓冲区 |

**FileResponse 的工作原理**：

```
浏览器请求: GET /raw/0/movie.mp4
  Header: Range: bytes=1048576-

服务端响应: 206 Partial Content
  Header: Content-Range: bytes 1048576-1073741823/1073741824
  Body: [文件偏移 1MB 处开始的数据]

底层实现: sendfile(fd_in, fd_out, offset, count)
  内核态直接传输，数据不经过用户空间
```

### 5.3 文件上传流式写入

**重构前（有问题）**：


### 6.2 路径穿越防护

**威胁模型**：攻击者通过 URL 中的 `../../../etc/passwd` 尝试逃逸出共享目录，访问服务器上的任意文件。

**重构前（有漏洞）**：

```python
# 字符串前缀判断 — 可被绕过
full = str(root / rel)
if not full.startswith(str(root)):
    return None  # 但 symbolic link 或编码差异可绕过
```

**重构后（安全）**：

```python
def resolve_path(self, share_idx: int, rel: str) -> Path | None:
    root = self.get_share_root(share_idx)
    if root is None:
        return None

    # 逐段解析路径，过滤 . 和空段
    parts = [p for p in unquote(rel).replace('\\', '/').split('/') if p and p != '.']

    current = root
    for part in parts:
        if part == '..':
            current = current.parent
        else:
            current = current / part

    # 关键：Path.is_relative_to() 检查是否在根目录内
    if not current.is_relative_to(root):
        return None

    return current
```

`Path.is_relative_to()` 是 Python 3.9+ 的标准方法，基于路径解析而非字符串匹配，无法被编码差异绕过。

### 6.3 登录限流

**策略**：

```
连续失败 1~4 次 -> 正常，记录失败次数
连续失败 5 次   -> 锁定 5 分钟（300 秒）
锁定期间访问    -> 返回 "N 秒后重试"
成功登录        -> 清除所有失败记录
30 分钟无活动   -> 自动清理 IP 记录
```

**防内存泄漏**：每次 `check()` 调用时触发 `_cleanup_expired()`，最多清理 50 条过期记录。这避免了恶意 IP 持续请求导致内存无限增长。

### 6.4 Cookie 鉴权

```
登录成功 -> Set-Cookie: token=<SHA256哈希>; HttpOnly; Max-Age=2592000; Path=/
                                                         (30天)

请求到达 -> 中间件读取 Cookie.token
  +-- 与 self.valid_token 匹配 -> 放行
  \-- 不匹配
       +-- API 路径  -> JSON 401 {"ok": false, "msg": "未登录"}
       \-- 页面路径  -> 302 重定向到 /login
```

`HttpOnly` 标志防止 JavaScript 通过 `document.cookie` 读取 token，降低 XSS 攻击的影响。

## 第八章 开发过程中遇到的问题与解决方案

### 8.1 移动端视频播放：600MB 文件无法秒开

**现象**：手机浏览器打开 600MB 的 MKV 视频，等待数十秒无响应。

**原因分析**：
- MKV 容器中视频编码为 H.264（手机兼容），但音频编码为 AC-3（手机不支持）
- 初始方案对整个视频做 `-c:v libx264` 全量重编码，600MB 文件需要数分钟

**解决方案**：三级转码策略链，优先尝试最快方案：

1. **H.264 + AAC** -> `-c copy`（全部复制，秒级）
2. **H.264 + 非AAC** -> `-c:v copy -c:a aac`（只转音频，快）
3. **非H.264** -> 先试策略 2，失败再 `-c:v libx264`（全量重编码）

**效果**：600MB MKV 从"等待数分钟"变为"秒开"，因为只重编码了音频轨。

### 8.2 转码视频暂停恢复从头开始

**现象**：转码后的视频在手机上点暂停再恢复，进度条跳回 0:00。

**原因分析**：
- `-c:v copy` 保留源文件的关键帧结构，如果源文件关键帧稀疏（如每 10 秒一个），手机播放器在恢复时找不到最近的关键帧，只能从头开始
- 全量重编码（`-c:v libx264`）可以插入密集关键帧解决此问题，但速度太慢

**解决方案**：用户选择了"速度优先"策略 — 接受暂停恢复从头开始的限制，换取秒级转码速度。这是经过权衡的有意设计决策，非 bug。

**记录在 CLAUDE.md 中**：
> 只转音频的视频（`-c:v copy`）暂停恢复可能从头开始，因为关键帧结构保留自源文件。用户偏好：优先速度，暂停问题可接受。

### 8.3 ffmpeg 进程死锁：stderr 管道缓冲区满

**现象**：ffmpeg 转码进程卡住不动，既不结束也不输出。

**原因分析**：
- 操作系统为子进程的 stdout/stderr 分配了固定大小的管道缓冲区（通常 64KB）
- ffmpeg 的 stderr 输出大量编码信息，如果不及时读取，缓冲区满后 ffmpeg 的 `write()` 系统调用会阻塞
- 主协程在等待 ffmpeg 结束，ffmpeg 在等待 stderr 被读取 — 死锁

**解决方案**：启动一个并发协程持续 drain stderr：

```python
async def drain_stderr(proc):
    """持续读取 stderr 防止管道缓冲区满导致 ffmpeg 死锁"""
    while True:
        line = await proc.stderr.readline()
        if not line:
            break
        # 可选：解析进度信息
```

在 `_handle_transcode()` 中同时 `await` 转码完成和 drain 协程：

```python
await asyncio.gather(
    proc.wait(),           # 等待 ffmpeg 结束
    drain_stderr(proc)     # 持续读取 stderr
)
```

### 8.4 XSS 注入漏洞：恶意文件名执行脚本

**现象**：如果文件名为 `<script>alert(1)</script>.mp4`，在浏览器中打开目录浏览页时会弹出 JavaScript alert 框。

**原因分析**：
- `pages.py` 的模板函数直接将文件名拼接进 HTML 字符串
- 浏览器将 `<script>` 标签解析为可执行代码

**解决方案**：在所有模板函数中，对用户可控字符串使用 `html.escape()` 转义：

```python
# 重构前
file_items += f'<div class="file-name">{name}</div>'

# 重构后
safe_name = html_mod.escape(name)
file_items += f'<div class="file-name">{safe_name}</div>'
```

转义后输出：`<div class="file-name">&lt;script&gt;alert(1)&lt;/script&gt;.mp4</div>`，浏览器显示为纯文本而非执行脚本。

**测试验证**：`test_pages.py` 中有 6 个专门的 XSS 测试用例，覆盖首页、浏览页、播放器、管理页的各种恶意输入场景。

### 8.5 路径穿越漏洞：字符串前缀判断可绕过

**现象**：通过构造特殊路径（如 `../../../etc/passwd`）可以访问共享目录之外的文件。

**原因分析**：
- 原始实现用 `str(resolved).startswith(str(root))` 判断路径是否在共享目录内
- 字符串比较不考虑路径规范化，某些编码方式可能绕过前缀检查

**解决方案**：改用 `Path.is_relative_to()` 方法：

```python
# 重构前（不安全）
if not str(current).startswith(str(root)):
    return None

# 重构后（安全）
if not current.is_relative_to(root):
    return None
```

`Path.is_relative_to()` 在路径语义层面进行比较，不受编码差异影响。

### 8.6 RateLimiter 内存泄漏：IP 记录永不清理

**现象**：服务运行数天后内存持续增长。

**原因分析**：
- `RateLimiter` 将每个失败 IP 的记录存储在字典中
- 原始实现没有清理机制，即使 IP 已经不再访问，记录也永久保留
- 长期运行后，大量僵尸 IP 记录占用内存

**解决方案**：
1. 用 `@dataclass _IPRecord` 替代裸 list，增加 `last_active` 时间戳
2. 每次 `check()` 调用时触发 `_cleanup_expired()`
3. 清理策略：`last_active` 超过 30 分钟的记录被移除
4. 单次最多清理 50 条（`_MAX_CLEANUP_PER_CHECK`），防止瞬时性能抖动

### 8.7 事件循环阻塞：同步 subprocess 卡死异步服务

**现象**：生成视频缩略图时，整个 HTTP 服务无响应数秒。

**原因分析**：
- `generate_thumbnail()` 使用同步 `subprocess.run()` 调用 ffmpeg
- 在 async handler 中调用同步阻塞函数会卡住整个事件循环
- ffmpeg 处理大文件可能需要 5~15 秒，期间所有其他请求被阻塞

**解决方案**：新增 `async_generate_thumbnail()`，使用 `asyncio.create_subprocess_exec()`：

```python
# 重构前（阻塞事件循环）
result = subprocess.run(args, timeout=15, ...)

# 重构后（非阻塞）
proc = await asyncio.create_subprocess_exec(*args, ...)
_, stderr = await proc.communicate()
```

同样，`probe_video_codec` 也从同步改为 `async_probe_video_codec`。

### 8.8 上传内存溢出：大文件全部读入内存

**现象**：上传 2GB 视频时，服务端进程内存暴涨到 2GB+。

**原因分析**：
- 原始实现用 `await field.read()` 一次性读取整个文件到内存
- 对于大文件，这会导致内存耗尽

**解决方案**：改为流式写入，每次只读 1MB：

```python
# 重构前（内存爆炸）
data = await field.read()
f.write(data)

# 重构后（内存恒定 1MB）
with open(dest, 'wb') as f:
    while True:
        chunk = await field.read_chunk(1024 * 1024)
        if not chunk:
            break
        f.write(chunk)
```

### 8.9 Windows 控制台编码崩溃：GBK 无法渲染中文/emoji

**现象**：`print("🎵 服务器启动")` 抛出 `UnicodeEncodeError`。

**原因分析**：
- Windows 中文版控制台默认使用 GBK 编码
- GBK 无法编码 emoji 和部分 Unicode 字符

**解决方案**：`safe_print()` 函数捕获 `UnicodeEncodeError`，将无法编码的字符替换为 `?`：

```python
def safe_print(msg: str):
    try:
        print(msg)
    except UnicodeEncodeError:
        print(msg.encode('utf-8').decode('gbk', errors='replace'))
```

重构后，所有模块改用 `logging` 模块输出日志（UTF-8 文件 + stdout），`safe_print` 保留为兼容层。

### 8.10 SyntaxError：nonlocal 变量声明顺序错误

**现象**：运行时报 `SyntaxError: name 'tmp_path' is used prior to nonlocal declaration`。

**原因分析**：
- 在 `_handle_transcode()` 的内部函数 `try_copy_then_encode()` 中，代码先使用了 `tmp_path` 变量，后声明 `nonlocal tmp_path`
- Python 语法要求 `nonlocal` 声明必须在变量使用之前

**解决方案**：删除 `nonlocal` 声明。`tmp_path` 是 `_handle_transcode` 的局部变量，内部函数可以直接读取（闭包捕获），不需要 `nonlocal`。只有需要**赋值修改**外层变量时才需要 `nonlocal`。

### 8.11 PDF 资源泄漏：异常时文档句柄未关闭

**现象**：大量 PDF 请求后，系统文件句柄耗尽。

**原因分析**：
- `read_pdf_page()` 使用 `fitz.open()` 打开 PDF 文档
- 原始实现用 `doc.close()` 手动关闭，但如果在 `open()` 和 `close()` 之间发生异常，文档句柄泄漏

**解决方案**：改用上下文管理器：

```python
# 重构前（异常时泄漏）
doc = fitz.open(path)
# ... 可能抛异常的操作 ...
doc.close()

# 重构后（自动关闭）
with fitz.open(path) as doc:
    # ... 即使抛异常也会自动关闭 ...
```

### 8.12 重复常量定义：VIDEO_EXTS 在两处各定义一次

**现象**：`utils.py` 和 `handlers.py` 各自定义了一份 `VIDEO_EXTS`，修改一处后另一处不同步。

**原因分析**：早期开发时在两个文件中分别定义了视频扩展名集合，缺乏统一管理。

**解决方案**：只在 `utils.py` 中定义一次，`handlers.py` 通过 import 引用：

```python
# handlers.py
from .utils import VIDEO_EXTS  # 不再重复定义
```

### 10.4 ffmpeg 转码参数速查

**策略 1：全复制（秒级）**

```bash
ffmpeg -i input.mkv -c copy -movflags +faststart output.mp4
```

**策略 2：只转音频（快）**

```bash
ffmpeg -i input.mkv -c:v copy -c:a aac -b:a 128k -movflags +faststart output.mp4
```

**策略 3：全量重编码（慢）**

```bash
ffmpeg -i input.mkv -c:v libx264 -preset fast -crf 26 -vf scale=1280:-2 \
  -c:a aac -b:a 128k -movflags +faststart output.mp4
```

**公共错误恢复参数**：

```bash
-fflags +discardcorrupt+genpts+igndts
-err_detect ignore_err
-avoid_negative_ts make_zero
-max_muxing_queue_size 4096
```

```bash
ffmpeg -ss 5 -i input.mp4 -vframes 1 -vf scale=320:-1 -y output.jpg
```
